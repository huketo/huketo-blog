## 하이퍼레저 패브릭 개요

하이퍼레저 패브릭은 **MSP(Membership Service Provider)** 라는 인증 관리 시스템에 등록된 사용자만이 블록체인에 참여가 가능한 허가형 프라이빗 블록체인입니다.  
하이퍼레저 패브릭의 참여자들은 비즈니스 목적에 알맞은 형태로 블록체인 플랫폼을 구축할 수 있습니다 => 블록체인 프레임워크

또한 **채널(Channel)** 이라는 개념을 도입해서 블록체인 참여자들 간의 프라이버시를 강화할 수 있습니다.

### 채널(Channel)

![채널그림](#)

모든 사용자가 동일한 원장을 가지고 모든 정보를 공유할 수 있을 뿐만 아니라, 비즈니스에 민감한 내용을 공유하고 싶은 참여자들 간에만 채널을 통해서 별도의 원장을 생성하여 정보를 공유할 수도 있습니다.

### 원장(Shared ledger)

![원장그림](#)

- **월드 스테이트(World State)** : 원장의 현재 상태  
  ex) 은행: 잔고에서 현재 가지고 있는 금액

- **블록체인(Blockchain)** : 원장의 전체기록  
  ex) 은행: 계좌를 만든 후 부터 현재까지의 모든 거래기록

### 체인코드(Chaincode)

하이퍼레저 패브릭에서 스마트 컨트랙트는 **체인코드(Chaincode)** 에 쓰여집니다. 체인코드는 기존의 스마트 컨트랙트와 같이 원장에 데이터를 읽고 쓰기 위해 사용될 수 있습니다.  
스마트 컨트랙트와의 차이점은 **시스템 체인코드(System Chaincode)** 라는 특수한 체인코드를 이용하여 블록체인 시스템 설정이 가능하다는 특징이 있습니다.

## 하이퍼레저 패브릭 특징

- 프라이버시와 기밀성
- 작업구간별 병렬 처리
- 체인코드
- 모듈화된 디자인

하이퍼레저 패브릭은 블록체인 참여자들 간에 **채널(Channel)** 개념을 도입하여 **프라이버시** 와 **기밀성** 을 제공

![데이터 처리과정](#) Excute -> Order -> Validation

트랜잭션 생성부터 합의하는 과정까지 단계별로 분리하여 처리

1. **Excute(실행)**: 트랜잭션을 실행하고 결괏값을 검증
2. **Order(정렬)**: 실행 단계에서 검증이 끝난 트랜잭션을 취합하여 순서에 맞게 정렬한 후 블록을 생성하는 작업을 수행
3. **Validation(검증)**: 블록에 포함된 모든 트랜잭션에 대한 결괏값 검증을 수행하고, 각종 디지털 인증서 등을 확인한 후 이상이 없을 시 최신블록을 업데이트

## 하이퍼레저 패브릭의 구성요소

### 피어(Peer)

**Peer**: 하이퍼레저 패브릭 블록체인을 구성하는 네트워크 노드 중 하나

- 블록체인의 중요요소 3가지: _분산원장_, _스마트 컨트랙트_, _합의_

Peer는 _분산원장_ 과 _스마트컨트랙트_ 를 관리

> 하이퍼레저 패브릭에서는 *스마트 컨트랙트*라는 용어 대신 _체인코드(Chaincode)_ 라는 용어를 사용합니다.

패브릭 참여자는 *peer*를 통해서만 분산원장과 체인코드에 접근할 수 있습니다. 필요에 따라서 *peer*는 복수 개의 분산원장과 체인코드를 가질 수 도 있습니다. 체인코드를 통해여 분산원장에 접근하여 데이터를 기록하고 읽어오는 데 사용됩니다. 분산원장에 접근가능한 체인코드가 없는 경우도 있지만 권장사양이 아닙니다.  
공식 문서에서는 최소 한 개 이상의 분산원장과 체인코드 호스팅을 권장하고 있습니다.

![peer 노드 네트워크 예시](#)

노드를 통해 데이터를 기록하거나 읽어올 수 있고, 만약 데이터를 기혹하는 작업을 요청한다면 [orderer]() 노드와 함께 합의 과정을 진행하여 분산원장을 업데이트 합니다.

### 체인코드(Chaincode)

블록체인의 핵심은 한번 기록된 데이터에 대한 위변조가 불가능한 *분산원장*에 있습니다.

하이퍼레저 패브릭에서는 *분산원장*에 데이터를 기록하거나 읽기 위해서 **체인코드(Chaincode)** 가 필요합니다. 주로 비즈니스 모델에 맞는 Dapp과 함께 개발되어 사용됩니다.

![잔액 확인 트랜잭션 생성](#)

#### 시스템 체인코드(System Chaincode)

퍼블릭 블록체인의 _스마트 컨트랙트_ 와 달리 하이퍼레저 패브릭에는 *시스템 체인코드*라는 특수한 체인코드가 존재합니다.  
_시스템 체인코드_ 는 애플리케이션 레벨에서 원장에 접근하는 체인코드와는 달리 하이퍼레저 패브릭의 시스템 레벨에서 수행되는 체인코드입니다. 시스템 체인코드는 하이퍼레저 패브릭 상에서 기본적으로 제공하는 체인코드이며, 5종류의 시스템 체인코드가 있습니다.

- **QSCC(Query System ChainCode)**: 블록체인의 저장된 데이터를 읽어올 때 사용
- **ESCC(Endorsement System ChainCode)**: 보증 정책을 담당
- **VSCC(Validation System ChainCode)**: 블록을 검증할 때 사용
- **CSCC(Configuration System ChainCode)**: 채널 설정 시 사용
- **LSCC(Lifecycle System ChainCode)**: 체인코드의 설치부터 인스턴스화까지 모든 일련의 과정을 수행하는 데 사용

| System Chaincode | 실행 방법                                                                                                |
| ---------------- | -------------------------------------------------------------------------------------------------------- |
| QSCC, CSCC, LSCC | 사용자에 의해 CLI 명령어로 실행                                                                          |
| ESCC, VSCC       | Endoring peer(트랜잭션의 보증을 담당하는 peer), Committing peer(블록에 대한 검증을 담당하는 peer)로 실행 |

#### QSCC(Query System ChainCode)

![QSCC 실행 예](#)

**QSCC**는 블록체인의 저장된 데이터를 읽어올 때 사용되는 시스템 체인코드입니다.

채널의 구성원은 CLI 명령어를 통해 _블록 번호_, _블록의 해시값_, _트랜잭션 ID_ 등 다양한 입력값을 통해 데이터를 읽어올 수 있습니다.

#### ESCC(Endorsing System ChainCode)

![ESCC 실행 예](#)

**ESCC**는 보증 정책을 담당하는 시스템 체인코드입니다.

*Endorsing peer*는 *ESCC*를 통해 사용자의 트랜잭션 실행 결괏값을 비교해 보고 결괏값이 올바르면 자신의 인증서를 통해 해당 트랜잭션 결괏값에 대한 보증을 합니다. 기본적으로 시스템에서 제공하는 _ESCC_ 체인코드를 사용하지만 필요에 따라서 사용자의 비즈니스 로직에 맞게 수정해서 사용할 수 있습니다.

#### VSCC(Vaildation System ChainCode)

![VSCC 실행 예](#)

**VSCC**는 블록을 검증할 때 사용되는 시스템 체인코드입니다.

Committing peer는 VSCC를 실행하여 트랜잭션의 Read/Write Set과 보증 정책에 맞게 Endorsing peer의 디지털 인증서의 존재 여부를 확인하는 작업을 수행합니다. 마찬가지로 ESCC 처럼 필요에 따라서 수정해서 사용할 수 있습니다.

#### CSCC(Configuration System ChainCode)

![CSCC 실행 예](#)

**CSCC**는 채널 설정 시 사용되는 시스템 체인코드입니다.

CSCC는 블록에 대한 정보를 읽어오거나 수정할 수 있으며, peer를 채널에 참여시키는 기능 등을 제공합니다.

#### LSCC(Lifecycle System ChainCode)

![LSCC 실행 예](#)

**LSCC**는 체인코드의 설치부터 인스턴스화까지 모든 일련의 과정을 수행하는 데 사용되는 체인코드입니다.

### Dapp

체인코드는 체인코드 그 자체만으로도 사용될 수 있지만, 대부분의 경우 비즈니스 모델에 맞는 **Dapp**과 함께 개발되어 사용됩니다.

하이퍼레저 패브릭에서는 다양한 종류의 **SDK**를 제공하고 있습니다. 개발자는 **SDK**를 통해서 트랜잭션을 생성하고 체인코드 함수를 불러오는 등 여러 기능을 좀 더 쉽게 개발할 수 있습니다.

체인코드의 기능은 크게 읽기(query) / 쓰기(write) 두 종류로 구분

#### Dapp을 통한 분산원장 데이터 읽기 과정

![Dapp을 통한 분산원장 데이터 읽기 과정](#)

1. 트랜잭션 생성 요청
2. peer와 연결: 인증과정을 통과한 후 연결
3. 체인코드 실행 요청: query 함수 호출
4. 체인코드 실행: 로컬 정상소에 저장되어 있는 분산원장의 데이터를 분산 애플리케이션에 전달
5. 결괏값 반환

#### Dapp을 통한 분산원장 업데이트 과정

![Dapp을 통한 분산원장 업데이트 과정](#)

1. 트랜잭션 생성 요청
2. peer와 연결
3. 체인코드 실행 요청
4. 체인코드 실행
5. 보증 허가된 트랜잭션 반환
6. 보증 허가된 트랜잭션 전달
7. 보증 허가된 트랜잭션을 최신 블록에 포함
8. 최신블록 전달

=> 분산원장에 데이터를 기록하는 작업은 peer간의 합의 과정이 필요, 모든 peer가 쓰기 과정에 참여

> **① 디지털 인증서와 트랜잭션 시뮬레이션 결괏값 전달**  
> 분산원장에 데이터를 기록하기 위해서는 보증 정책을 충족시켜야 합니다. 보증조건을 충족시키면 인증서와 결괏값을 Dapp에 전달합니다.

> **⑥, ⑦, ⑧ Orderer 가 트랜잭션을 모아 peer들에게 최신블록 전달**  
> 1. Dapp은 트랜잭션 결괏값과 디지털인증서와 함께 트랜잭션을 Orderer노드로 전송합니다.
> 2. Orderer노드는 자신이 속한 네트워크에서 발생한 모든 트랜잭션을 수신하게 됩니다.
> 3. Orderer는 수신한 트랜잭션을 순서에 맞게 정렬하여 블록체인의 최신블록을 생성하고, 생성한 블록을 자신이 속한 네트워크의 모든 peer에게 전달합니다.
>
> *Orderer는 트랜잭션을 순서에 맞게 정렬하여 블록을 생성만 할 뿐 트랜잭션의 내용을 확인하고 검증하지는 않습니다.*

### 보증정책(Endorsement Policy)
